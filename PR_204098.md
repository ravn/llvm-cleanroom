Fixes #204098.

## Summary

On the Z80 it is very expensive to do 16-bit arithmetic, when 8-bit would do.  This bug also applies to AVR which is a support target in-tree.  

This PR allows `TruncInstCombine` to consider 8-bit exit conditions as valid narrowing candidates, even when the loop body is in 16-bit arithmetic, like for K&R-style loops. The fix is to treat `icmp` instructions as a new kind of safe outside-graph user, as long as their operands have KnownBits proving they fit in the narrow type.

Signed predicates and `(and X, Const)` outside-users are deliberate
follow-ups; both extend the same plumbing.

I have done this using Claude, but have tried to do this as clear and
simple from the issue as I could.

The text below explains things in more detail as written by Claude. I
have reviewed it but not rewritten it, as it was clear enough on its
own.

## Detailed analysis (by Claude `claude-opus-4-7`, reviewed by Thorbjørn)

### The bail

`TruncInstCombine::getBestTruncatedType` rejects the whole rewrite when
any in-graph instruction has a user that is (a) not in the graph, (b)
not the current `TruncInst`, and (c) not a `ZExt`/`SExt`. The
`Zext/Sext` exception exists because those casts can be folded away by
the narrowing itself. Every other shape forfeits the rewrite.

That's conservative-correct but blocks a very common shape: a loop
whose body computes in `iN` arithmetic and whose loop-exit compare
uses one of the in-graph values. Per the bug report, on AVR the
classic gf_log byte-loop pattern is ~30 % slower and ~31 % larger than
the achievable codegen because of this single bail.

### Admission

Add a new outside-user shape: an `ICmpInst` whose predicate is
equality or unsigned, **and** whose operands both have KnownBits
proving fit in the narrow type. With the narrow-bit-width validated,
the rewriter replaces the wide icmp with an icmp of truncated
operands.

The check has to be deferred. The outside-user loop runs at the top
of `getBestTruncatedType`, before `getMinBitWidth()`. So the loop just
collects candidates; a validation pass after `MinBitWidth` is known
applies the KnownBits gate at the correct width.

### Why the in-graph KnownBits gate is load-bearing

The wide value's narrowed form is `trunc(V, NarrowBits)`. If the wide
value's full bit-pattern exceeds `2^NarrowBits - 1`, the truncated
form differs from the wide form, and the icmp result can change.
KnownBits gives the soundness check: `getMaxValue().getActiveBits() <=
MinBitWidth` for every operand.

Concrete witness for what happens without the gate (paraphrased from
the bug body): `%add = add i32 %x, 1; icmp ult i32 %add, (and y,
255)` with `%x = 65636`. Low 16 bits of `%add` = 100, wider value =
65637. With `(and y, 255)` = 200, the wide compare is **false**
(65637 > 200) but a naïve narrowed compare on the low 16 bits would
be **true** (100 < 200). The KnownBits gate is what keeps the rule
sound; it would catch this case because `getMaxValue` of `%add` is
`UINT32_MAX`.

### Rewrite

`ReduceExpressionGraph` gets a new step right after the main rewrite
loop has populated `Info.NewValue` for every in-graph instruction and
the phi-incoming hookup is done — but **before** the phi-erase RAUW
loop that replaces old phis with poison. A still-wide icmp referencing
an in-graph phi would otherwise capture poison.

For each admitted icmp, a small local helper produces the narrow form
of each operand: `getReducedOperand` semantics for in-graph and
`Constant` operands; for anything else, emit a `trunc V to NarrowTy`
at the icmp's location. The new `icmp <pred> L_narrow, R_narrow` takes
the old icmp's name and replaces it.

### Predicate scope cut for this PR

Equality (`eq`/`ne`) and unsigned (`ult`/`ule`/`ugt`/`uge`) only.
Signed predicates need a different fit-bits rule (`NarrowBits - 1`
unsigned, or signed-range representability) and add their own
reviewer surface, so they're a follow-up that extends the same
candidate-collection plumbing.

### `(and X, Const)` outside-users — also a follow-up

The bug body describes a second outside-user shape: an `and` whose
constant fits the narrow type. That admission is sound regardless of
the in-graph KnownBits because the mask discards high bits
unconditionally, but it needs a different rewriter (the and's result
is wide, not `i1`). Treating it as a separate PR keeps this PR's
soundness story focused on the icmp/KnownBits case and lets the
follow-up build on this PR's collection mechanism.

### Testing

New: `llvm/test/Transforms/AggressiveInstCombine/trunc_icmp_user.ll`
covers:

- equality icmp on two in-graph values (both KnownBits-fit) — narrows;
- equality icmp on an in-graph value vs a small `ConstantInt` — narrows;
- unsigned icmp on an in-graph value vs a small `ConstantInt` — narrows;
- equality icmp on an in-graph value whose KnownBits **don't** fit
  (`add %ingraph, 1` reaches 256) — must reject (soundness gate);
- signed icmp on an in-graph value — must reject (out-of-scope
  predicate);
- non-icmp non-ext outside user (store) — must still bail.

Each positive case fails on unpatched `opt` and passes with the patch.
Each negative case has identical output pre- and post-patch.

One existing test, `trunc_select_cmp.ll`, has one sub-case
(`cmp_select_unsigned_const_i16Const`) whose CHECK lines updated:
the icmp is now admitted, the graph narrows further (to `i16`
instead of staying at `i32`), and the constant `32768` truncates
losslessly to `i16 -32768`. Semantics preserved.

Lit results:
- `Transforms/AggressiveInstCombine` — 50/50 pass (3 target-conditional
  `Unsupported`, same as baseline).
- Broader sweep over
  `Transforms/{InstCombine,AggressiveInstCombine,PhaseOrdering}` —
  1991 of 2078 pass, 86 `Unsupported`, 1 `XFAIL`, **0 regressions**
  beyond the one CHECK-line update above.
